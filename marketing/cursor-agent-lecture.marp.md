# CursorとAIエージェントの社内活用

## タスクを分類し、依頼し、成果物をレビューする

---

# 今日の目的

- Cursorを「AI付きエディタ」ではなく、業務を進める作業環境として理解する
- エージェントに任せやすいタスクを分類できるようにする
- GitHub Issue / Pull Request と組み合わせた運用を理解する
- 社内で再利用できる依頼文の型を持ち帰る

---

# Cursorでできること

- ファイルを読みながら相談する
- 文章やコードを編集する
- 複数ファイルをまたぐ作業をエージェントに任せる
- GitHub上のIssueやPull Requestと連携して成果物を確認する

現実を見るんだ。AIは魔法じゃない。だが、正しく使えば強力な相棒になる。

---

# チャット・編集・エージェントの違い

| 機能 | 向いていること | 成果物 |
| --- | --- | --- |
| チャット | 質問、相談、要約 | 回答、方針 |
| 編集 | 開いているファイルの修正 | ファイル差分 |
| エージェント | 複数手順の作業 | コミット、Pull Request |

---

# エージェントに任せやすい仕事

1. 調査・リサーチ
2. 文章作成・編集
3. 資料・講義コンテンツ作成
4. GitHub運用支援
5. データ整理・分析補助
6. 業務フロー改善
7. コード・自動化支援

---

# 1. 調査・リサーチ

## 目的

意思決定の前に、必要な情報を集めて整理する。

## 依頼例

- 競合企業のWebサイト構成を比較する
- 過去のIssueやPull Requestから類似事例を抽出する
- キャンペーンの訴求軸を整理する

---

# 2. 文章作成・編集

## 目的

社内外向けの文章を作成、改善、再構成する。

## 依頼例

- 社内告知文を作る
- 営業メールのトーンを調整する
- FAQを顧客向けの表現に直す

---

# 3. 資料・講義コンテンツ作成

## 目的

説明会、研修、社内勉強会で使う材料を作る。

## 依頼例

- Cursor講義のアジェンダを作る
- ハンズオン課題を作る
- 受講者向けチェックリストを作る

---

# 4. GitHub運用支援

## 目的

Issue、Pull Request、レビューを使った業務管理を補助する。

## 依頼例

- Issueテンプレートを作る
- PR説明文を改善する
- 変更内容をレビュー観点で要約する

---

# 5. データ整理・分析補助

## 目的

表、CSV、ログ、アンケートなどを判断しやすい形にする。

## 依頼例

- アンケート回答をカテゴリ別に整理する
- 顧客フィードバックをテーマ別に分類する
- 月次レポートの要点をまとめる

---

# 6. 業務フロー改善

## 目的

繰り返し作業を整理し、手順化または自動化する。

## 依頼例

- レポート作成手順を標準化する
- キャンペーン開始前チェックリストを作る
- 承認フローを図解する

---

# 7. コード・自動化支援

## 目的

軽量なスクリプトや設定ファイルで作業を効率化する。

## 依頼例

- Markdownのリンク切れを検査する
- CSVを整形するスクリプトを作る
- GitHub Actionsの設定を作る

---

# タスクとプロジェクトの違い

| 判断項目 | タスクでよい | プロジェクト化する |
| --- | --- | --- |
| 成果物 | 1つの文書や修正 | 複数の成果物 |
| 関係者 | 依頼者のみ | 複数部署 |
| 確認方法 | 依頼者が確認 | レビューや承認が必要 |
| 継続性 | 一度で終わる | 継続運用がある |

---

# プロジェクト分類

- マーケティング基盤整備
- コンテンツ制作支援
- 業務改善・自動化
- 営業・カスタマーサクセス支援
- 社内教育・ナレッジ共有

---

# 依頼文の基本形

```text
目的:
背景:
対象ファイルまたは対象情報:
成果物:
制約:
確認方法:
```

曖昧な依頼は、曖昧な成果しか返さない。任務は具体化するんだ。

---

# よい依頼の例

```markdown
## 目的
社内向けCursor講義で使うサンプルIssueを作る。

## 依頼内容
- マーケティング業務で使える依頼例を10件作る
- 難易度を beginner / intermediate / advanced に分ける
- 各依頼に期待成果物とレビュー観点を付ける

## 成果物
docs/sample-agent-tasks.md

## 確認方法
受講者が自分の業務へ置き換えられるか確認する。
```

---

# 悪い依頼の例

```text
いい感じに資料を作って。
```

## 問題点

- 目的がない
- 誰向けか分からない
- 成果物の形式が分からない
- 完了条件がない

口だけではだめだ。AIにも任務説明が必要だ。

# Pull Requestで確認する

## 見るべきポイント

- 依頼した成果物が作られているか
- 変更範囲が広がりすぎていないか
- 機密情報や不適切な表現がないか
- レビュー観点に対して説明できるか
- 必要な検証が行われているか

---

# 人間が判断すること

- 事実として正しいか
- 会社の方針やブランドに合っているか
- 顧客や社員に誤解を与えないか
- 公開してよい情報か
- 最終的に採用するか

エージェントは作業者だ。判断者は人間である。

# チェックリスト

- [ ] 目的が一文で説明できる
- [ ] 成果物の形式が明確である
- [ ] 変更してよい範囲が明確である
- [ ] 参考情報がまとまっている
- [ ] 完了判断の基準がある
- [ ] 人間が判断すべき点を残している

---

# 社内運用の原則

- 小さく依頼する
- 成果物を明確にする
- GitHub Issueで管理する
- Pull Requestで確認する
- 人間が最終判断する
- 成功例と失敗例をナレッジ化する

---

# このCursorアカウントでの実例

## 本リポジトリで確認できるもの

- **amazon-eco**: アマゾン受け入れに伴う基盤変更（親リポジトリの業務区分）

講義では「自分の環境では何をしてきたか」を話すと説得力が増える。

- 事実ベースで書く

# まとめ

- Cursorは業務を進めるための作業環境である
- エージェントには分類された具体的な任務を渡す
- Issueで依頼し、PRで成果物を確認する
- 判断責任は人間に残る

戦場に英雄はいない。だが、準備したチームは生き残る。
